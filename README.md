# Домашнє завдання до Теми Поведінкові патерни: Ланцюжок відповідальностей та Посередник

### Опис завдання

Мета домашнього завдання — закріпити знання поведінкових патернів, **Ланцюжок відповідальностей** та **Посередник**, у сценарії обробки даних наближеному до реальності.

Ви реалізуєте систему, яка приймає масив записів у форматі `JSON`, проводить їхню валідацію, обробку та зберігає результати у відповідні файли залежно від формату.

Ваша задача — реалізувати обробник вхідних записів у форматі `.json`, кожен з яких має поле `type` та специфічну структуру. Дані необхідно передати через ланцюг обробників, який виконує валідацію та підготовку до збереження. Залежно від типу, результат зберігається у відповідний файл через централізованого посередника. У разі помилки — запис фіксується у файл відхилених `rejected.jsonl`.

Тож весь процес організовано у два рівні:

- Ланцюжки відповідальності відповідають за обробку даних різних типів: `access_log`, `transaction`, `system_error`;
- Посередник централізує процес збереження та маршрутизації результатів: успішні та помилкові записи.

Це завдання моделює практичну утиліту, яка аналізує структуровані дані, фільтрує помилки та зберігає звіти у форматі `.json`, `.csv` та `.jsonl`.

## Завдання

Структура проєкту

```
/
├── data/
│   └── records.json                 # Вхідний файл з необробленими записами
├── chain/
│   ├── AbstractHandler.ts           # Базовий клас для ланцюга відповідальностей
│   ├── handlers/
│   │   ├── TimestampParser.ts       # Парсинг timestamp у формат ISO
│   │   ├── UserIdValidator.ts       # Перевірка коректності userId
│   │   ├── IpValidator.ts           # Перевірка валідності IP-адреси (IPv4)
│   │   ├── AmountParser.ts          # Парсинг числових значень amount
│   │   ├── CurrencyNormalizer.ts    # Нормалізація валютного поля (ISO-формат)
│   │   ├── LevelValidator.ts        # Валідація рівня помилки (error, warning, info)
│   │   └── MessageTrimmer.ts        # Обрізання пробілів у повідомленні
│   └── chains/
│       ├── AccessLogChain.ts        # Ланцюг обробки access_log
│       ├── TransactionChain.ts      # Ланцюг обробки transaction
│       └── SystemErrorChain.ts      # Ланцюг обробки system_error
├── mediator/
│   ├── ProcessingMediator.ts        # Центральний посередник для збереження результатів
│   └── writers/
│       ├── AccessLogWriter.ts       # Збереження access_log у CSV
│       ├── TransactionWriter.ts     # Збереження transaction у JSON
│       ├── ErrorLogWriter.ts        # Збереження system_error у JSONL
│       └── RejectedWriter.ts        # Відхилені записи з помилками у JSONL
├── output/                          # Директорія для вихідних файлів
├── models/
│   └── DataRecord.ts                # Структура одного запису
└── main.ts                          # Точка входу для запуску
```

Вхідні дані — файл `records.json` містить масив записів.

## Як працює проєкт

Проєкт реалізовано з використанням двох поведінкових патернів:

- `Ланцюжок відповідальностей` (`Chain of Responsibility`) — кожен тип запису обробляється своїм ланцюгом обробників у папці `src/chain`. Кожен обробник перевіряє або перетворює частину запису, після чого передає результат далі. Якщо валідація не проходить, ланцюг викидає помилку.
- `Посередник` (`Mediator`) — центральний клас `src/mediator/ProcessingMediator.ts` управляє маршрутизацією оброблених записів в різні writer-и. Успішні записи передаються до відповідного writer-а, а помилкові — до `RejectedWriter`.

## Що реалізовано

- `access_log` — перевірка `timestamp`, `userId` та `ip`; результат зберігається в `src/output/access_logs.json`.
- `transaction` — перевірка `timestamp`, перетворення `amount` у число, нормалізація `currency` в верхній регістр; результат зберігається в `src/output/transactions.csv`.
- `system_error` — перевірка `timestamp`, валідація `level`, обрізка `message` до 255 символів; результат зберігається в `src/output/errors.jsonl`.
- `RejectedWriter` записує всі невдалі записи з описом помилки у `src/output/rejected.jsonl`.
- У `src/main.ts` додано підрахунок оброблених і відхилених записів, а також вивід короткого звіту в консоль.

## Як запускати проєкт

1. Встановіть залежності (якщо потрібно):

```bash
npm install
```

2. Запустіть застосунок:

```bash
npx ts-node src/main.ts
```

3. Після виконання перевірте директорію `src/output/` для створених файлів і короткий звіт у консолі:

```bash
[INFO] Завантажено записів: 19
[INFO] Успішно оброблено: 9
[WARN] Відхилено з помилками: 10
[INFO] Звіт збережено у директорії src/output/
```

## Формат вихідних файлів

- `src/output/access_logs.json` — масив об'єктів.
- `src/output/transactions.csv` — CSV з заголовком `timestamp,amount,currency`.
- `src/output/errors.jsonl` — JSONL-потік для `system_error`.
- `src/output/rejected.jsonl` — JSONL-потік з полями `record` і `error`.

## Як додати новий тип запису

1. Додайте новий тип у `src/models/DataRecord.ts` як інтерфейс і додайте його до об'єднання `DataRecord`.
2. Створіть ланцюг обробників у `src/chain/chains/`, використовуючи базовий клас `AbstractHandler`.
3. Реалізуйте потрібні обробники в `src/chain/handlers/` — кожен обробник має метод `process(record)`, який повертає оновлений запис або кидає помилку.
4. Додайте writer у `src/mediator/writers/`, який реалізує методи `write()` та `finalize()` для збереження результатів.
5. Зареєструйте writer у `src/mediator/ProcessingMediator.ts` і додайте відповідність у `writerMap`.
6. Додайте новий тип до `handlerMap` у `src/main.ts`, щоб для записів нового типу будувався потрібний ланцюг.
